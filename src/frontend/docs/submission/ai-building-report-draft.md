## 1. 기획

### 1-1. 설정한 사용자는 누구이며, 해결하려는 구체적인 문제점/불편함은 무엇인가요?

이 프로젝트의 핵심 사용자는 크게 두 층으로 나뉜다.

첫째는 **한국사를 공부하는 학생**이다. 현재 시험 대비형 학습 서비스는 대체로 "몇 문제를 맞았는지", "오답이 몇 개인지", "오늘 풀어야 할 문제가 몇 개인지" 정도는 잘 보여주지만, 학생이 **한국사 전체 흐름 안에서 정확히 어디에서 막혔는지**는 잘 보여주지 못한다. 한국사는 단순히 사건명이나 연도 하나를 외운다고 해결되는 과목이 아니라, `시대 -> 흐름 -> 사건 -> 개념 -> 문제`가 연결되어 있어야 이해가 된다. 그런데 기존 서비스에서는 오답이 단순히 "틀린 문제"로 남기 때문에, 학생 입장에서는 "왜 틀렸는지", "어떤 흐름이 끊겨서 틀렸는지", "오늘 무엇부터 복원해야 하는지"를 스스로 판단하기 어렵다. 그 결과 복습은 자주 하지만 같은 유형에서 다시 흔들리고, 시대 간 연결이 약해져 암기 피로도만 커지는 문제가 반복된다.

둘째는 **학생의 이해 누수를 관리해야 하는 선생님**이다. 선생님은 단순히 성적표를 보는 사람이 아니라, 학생이 어디서 이해를 놓쳤고 어떤 설명이 전달되지 않았는지를 파악해야 한다. 그러나 실제 수업과 시험 운영에서는 학생별 오답을 모두 손으로 해석해야 하고, "이 학생이 어떤 개념 연결을 놓쳤는지", "여러 학생에게 공통으로 새는 관계축이 무엇인지", "어떤 설명을 다시 해줘야 하는지"를 빠르게 파악하기 어렵다. 즉, 현재의 불편함은 **학생에게는 구조가 보이지 않는 학습**, **선생님에게는 재설명 우선순위가 보이지 않는 운영**이라고 정리할 수 있다.

이 프로젝트는 이 두 문제를 동시에 해결하려고 한다. 학생에게는 단순 문제 풀이 앱이 아니라 **한국사 흐름 복원형 학습 OS**를 제공하고, 선생님에게는 학생별 이해 누수와 관계망 수정 지점을 읽을 수 있는 **교육 운영 대시보드**를 제공한다. 즉, "몇 개 틀렸는가"보다 "어디서 연결이 끊겼는가"를 드러내는 것이 이 서비스의 본질이다.

### 1-2. 문제를 해결하기 위한 솔루션의 핵심 기능은 무엇인가요?

이 솔루션의 핵심 기능은 학생용 학습 루프와 선생님용 운영 루프가 하나의 구조 위에서 연결되도록 설계된 점이다. 현재 프론트엔드 기준으로는 다음과 같은 기능이 핵심으로 구현되어 있다.

#### A. 학생용 핵심 기능

1. **`/today` 오늘의 학습 진입면**
   - 오늘 먼저 봐야 할 중심 시대와 핵심 연결을 보여준다.
   - 학생이 현재 전체 한국사 시간축 중 어디를 공부하고 있는지 시각적으로 인지하게 만든다.
   - 단순 할 일 목록이 아니라, "오늘 복원해야 할 흐름"을 먼저 판단하게 한다.

2. **`/solve` 문제 풀이 적용면**
   - 문제를 푸는 행위를 단순 클릭이 아니라 "판단" 과정으로 다루도록 설계했다.
   - 현재 선택 상태, 배점, 진행 상태 등을 구조적으로 보여주고, 제출 후에는 해설과 연결된 복습 흐름으로 이어진다.

3. **`/notes` 오답 기록 보관실**
   - 오답을 단순 저장하지 않고, "오답 진단", "바로잡은 흐름", "기억 포인트"로 재구성한다.
   - 학생이 틀린 문제를 다시 볼 때, 정답 그 자체보다 왜 끊겼는지를 먼저 이해하도록 돕는다.

4. **`/review` 복습 회복실**
   - 복습 큐를 단순 반복 문제가 아니라, "기억을 회복해야 할 흐름" 중심으로 제시한다.
   - 오답노트와 문제풀이를 분리하지 않고, 복습 후 바로 문제 적용으로 이어지게 만든다.

#### B. 선생님용 핵심 기능

1. **`/admin` 운영 대시보드**
   - 학생 누수, 반복 실패, 관계망 편집 후보를 한 화면에서 볼 수 있는 교사용 운영면이다.
   - 단순 관리 페이지가 아니라 "어디를 먼저 개입해야 하는가"를 읽는 판단 대시보드 역할을 한다.

2. **`/admin/students` 학생 개입 운영실**
   - 학생별 개입 우선순위, 끊긴 흐름 수, 밀린 복습, 추천 조치를 콘솔형 레이아웃으로 제공한다.
   - 왼쪽에는 개입 큐, 오른쪽에는 선택 학생 상세 진단 패널을 두어 운영툴처럼 즉시 판단이 가능하도록 구성했다.

3. **`/admin/ontology` 관계망 편집실**
   - 선생님이 개념 노드와 관계축을 직접 수정하고, 어떤 구조가 학생 이해 누수와 연결되는지 판단하는 작업면이다.
   - 자동 추출이 출발점이라면, 최종 편집 권한은 선생님이 가진다는 전제를 반영했다.

#### C. 공통 시스템 기능

1. **한국사 학습 OS 테마 시스템**
   - 학생용 `/today`, `/notes`, `/review`, `/solve`에 공통으로 적용되는 `Time Descent` 기반 UI 시스템을 구축했다.
   - 한국사 느낌을 장식으로 만들지 않고, 기록성, 시간축, 복원감, 정갈한 위계로 번역했다.

2. **공용 프리미티브 및 토큰 구조**
   - `FrameLabel`, `SectionTitle`, `HeroShell`, `BottomRouteNav`, `RecordCard`, `StepCard` 등 재사용 가능한 UI 뼈대를 공용화했다.
   - 이를 통해 페이지가 늘어나도 같은 세계관으로 확장할 수 있게 했다.

3. **학생-선생님 구조 연결**
   - 학생에게는 "끊긴 연결", 선생님에게는 "전달 실패 지점"이라는 서로 다른 표현을 쓰되, 실제로는 같은 구조 문제를 보고 있다는 철학을 유지했다.

정리하면, 이 서비스의 핵심 기능은 "문제를 맞히게 하는 기능"이 아니라 **한국사 개념 전달과 이해 누수를 구조적으로 드러내고, 학생은 복원하고 선생님은 개입할 수 있게 하는 기능**이다.

### 1-3. 이 솔루션이 도입되었을 때 기대되는 개선점이 무엇인가요?

이 솔루션이 도입되면 가장 먼저 기대되는 개선점은 **학생의 복습 효율이 올라간다**는 점이다. 기존에는 학생이 오답노트와 문제풀이를 따로 보고, 무엇을 먼저 복습해야 하는지 스스로 판단해야 했다. 하지만 이 서비스에서는 중심 시대, 끊긴 연결, 복습 순서, 기억 포인트가 한 흐름으로 이어지기 때문에 학생이 "오늘 무엇을 먼저 정리해야 하는가"를 훨씬 빠르게 이해할 수 있다. 이는 단순 학습량 증가보다, 같은 시간을 쓰더라도 **재실패를 줄이고 회독 효율을 높이는 방향**으로 개선될 가능성이 높다.

두 번째 개선점은 **오답의 질이 바뀐다는 점**이다. 기존에는 오답이 문제 단위로만 쌓여서 "틀린 개수"는 알 수 있어도 "왜 반복해서 틀리는지"를 설명하기 어려웠다. 이 서비스는 오답을 `기록`, `진단`, `바로잡은 흐름`, `기억 포인트` 단위로 다루기 때문에, 학생은 자신의 약점을 더 구조적으로 이해하게 된다. 즉, 오답이 단순 저장물이 아니라 **이해 회복의 도구**로 바뀐다.

세 번째는 **선생님의 개입 품질이 높아진다**는 점이다. 선생님 입장에서는 학생별 누수와 공통 누수 축을 빠르게 확인할 수 있으므로, 막연히 다시 설명하는 것이 아니라 **가장 효과적인 재설명 순서**를 정할 수 있다. 또한 관계망 편집과 학생 누수 분석이 연결되기 때문에, 단순히 학생 개인 문제를 보는 것이 아니라 "이 구조 자체가 잘못 전달되고 있는가"라는 더 상위 레벨의 판단이 가능해진다. 이는 수업 준비와 피드백의 품질을 동시에 높일 수 있다.

네 번째는 **서비스 유지보수성과 확장성 개선**이다. 현재 프론트엔드는 학생 탭과 선생님 탭이 각각 독립된 화면처럼 보이지만, 실제로는 공용 디자인 토큰과 프리미티브를 바탕으로 확장 가능한 구조로 정리되어 있다. 이는 향후 `timeline`, `archive`, `focus`, `insight` 같은 새 탭을 추가하더라도 화면이 제각각 흩어지지 않도록 해준다. 즉, 이번 프로젝트는 단순히 "화면 몇 개 만든 수준"이 아니라, 이후 확장 가능한 **제품형 프론트 시스템**으로 진화할 기반을 확보했다는 점에서도 의미가 있다.

마지막으로, 교육 현장에서 기대되는 변화는 **정답 중심 학습에서 구조 중심 학습으로의 이동**이다. 학생은 어디서 끊겼는지 볼 수 있고, 선생님은 무엇을 다시 설명해야 하는지 볼 수 있으므로, 문제 풀이 결과가 단순 채점으로 끝나지 않고 다시 수업과 복습 설계로 환류된다. 이것이 도입 후 가장 큰 질적 개선점이라고 본다.

## 2. AI 활용 전략

### 2-1. 이 프로젝트에서 사용할 AI 도구와 모델은 무엇이며, 선택한 이유는 무엇인가요?

이 프로젝트는 화면 수가 많고, 학생/선생님 UI가 서로 다른 역할을 가지면서도 같은 제품 철학을 공유해야 하기 때문에, 단순 자동완성 수준의 AI보다 **코드베이스를 이해하고 구조 단위로 리팩터링할 수 있는 AI 도구 조합**이 필요하다. 본 프로젝트에서 사용할 AI 도구 전략은 다음과 같다.

#### 1. Codex 계열 코딩 에이전트

- 역할: 주 구현 에이전트
- 사용 목적: 다중 파일 리팩터링, 공용 프리미티브 추출, 레이아웃 수정, 오류 수정, lint/서버 검증
- 선택 이유:
  - 현재 프로젝트는 `/today`, `/notes`, `/review`, `/solve`, `/admin`, `/admin/students`, `/admin/ontology`처럼 여러 화면이 동시에 연결되어 있다.
  - 단일 파일 수정이 아니라 "한 화면의 개선이 다른 화면에 어떤 영향을 주는지"를 보면서 작업해야 하므로, 터미널 기반으로 코드 검색, 수정, 검증을 한 흐름 안에서 처리할 수 있는 에이전트가 적합하다.
  - 공용 UI 프리미티브 추출, 경로별 라우팅 정리, 하단 도킹 구조 수정, 디자인 토큰 공용화 같은 작업은 단순 자동완성보다 에이전트형 도구가 효율적이다.

#### 2. Cursor 같은 IDE 내 보조 AI

- 역할: 보조 구현/미세 반복 작업 도구
- 사용 목적: 작은 범위의 스타일 수정, 텍스트 카피 제안, 컴포넌트 인라인 수정, 빠른 실험
- 선택 이유:
  - 디자이너나 프론트엔드 개발자가 화면 단위로 빠르게 시도해보는 작업에는 IDE 안에서 바로 수정/미리보기 가능한 도구가 유리하다.
  - 긴 리팩터링보다는 특정 카드, 버튼, 카피, Tailwind 클래스 다듬기 등에 적합하다.
  - 팀 협업 과정에서 "큰 구조는 에이전트가, 작은 반복은 IDE 보조가" 나눠 쓰면 생산성이 높다.

#### 3. Claude Code 또는 Claude 계열 장문 추론 도구

- 역할: 상위 설계/리뷰 보조 도구
- 사용 목적: UX 방향성 정리, 대안 비교, 시스템 설계 문서 초안, 리팩터링 전 구조 검토
- 선택 이유:
  - 이 프로젝트는 단순 CRUD가 아니라 "학습 경험과 운영 경험을 어떻게 구조화할 것인가"가 중요하다.
  - 따라서 구현 전에 정보 구조, 화면 역할, 탭별 감정선, 디자인 철학을 긴 문맥으로 정리해주는 보조 도구가 유용하다.
  - 특히 학생 화면과 선생님 화면의 역할 분리, 한국사 서비스다운 질감, 테마 DNA 문서화 등은 장문 추론형 도구와 궁합이 좋다.

#### 4. 모델 선택 기준

이 프로젝트는 작업을 다음과 같이 나누어 모델을 선택한다.

- **고성능 추론 모델**
  - 사용 위치: 시스템 설계, 공용 컴포넌트 추출, 복잡한 레이아웃 변경, 전면 리팩터링
  - 이유: 여러 파일을 동시에 읽고 일관성 있게 바꾸는 작업에는 높은 추론 능력이 필요하다.

- **경량 모델 또는 IDE 보조 모델**
  - 사용 위치: 반복적인 텍스트 변경, 작은 CSS 수정, 카피 수정, 보일러플레이트 작성
  - 이유: 모든 작업에 최고 사양 모델을 쓰면 비용과 토큰이 과하게 늘어나기 때문이다.

즉, 이 프로젝트의 AI 도구 선택 이유는 단순히 "잘 써줘서"가 아니라, **구조적 리팩터링, 반복 보조, 장문 설계 검토를 서로 다른 역할로 분리해 생산성을 높이기 위해서**다.

### 2-2. 각 AI 도구별 활용 전략을 작성해주세요.

이 항목에서는 도구별 페르소나, 핵심 전략, 제약 사항, 에이전트 구성 방식, 데이터 흐름을 기준으로 정리한다.

#### A. Codex 계열 코딩 에이전트 활용 전략

**페르소나**
- "프론트엔드 구현 책임자 + 코드 리팩터링 담당자"

**핵심 전략**
- 화면 하나를 예쁘게 만드는 것보다, 여러 화면이 같은 설계 원칙 위에서 동작하게 만드는 데 집중한다.
- 페이지별 요구사항을 먼저 문서로 잠그고, 그다음 공용 컴포넌트/토큰으로 추출한 뒤, 각 페이지에 재적용한다.
- 구현 후에는 반드시 lint, 로컬 서버 응답, 실제 렌더링 기준으로 검증한다.

**주요 담당 작업**
- `/today`, `/notes`, `/review`, `/solve` UI 구조 리팩터링
- `history-ui` 공용 프리미티브 정리
- 하단 도킹 네비/액션 바 겹침 수정
- 학생 앱과 선생님 앱의 레이아웃 정리
- `admin/students` 콘솔형 2패널 구조화
- 제출용 문서 및 메모리 문서 생성

**제약 사항**
- 공용 프리미티브를 깨는 방향의 중복 코드 생성 금지
- "화려한 장식"보다 정보 구조 우선
- 학생 화면과 선생님 화면의 역할 차이를 유지
- 수정 후 검증 없는 마무리 금지

**에이전트 구성 방식**
- 메인 에이전트 1개가 구조 변경과 코드 통합을 담당
- 필요 시 보조 에이전트 또는 보조 도구는 탐색/비교 용도로만 사용
- 최종 반영과 검증은 반드시 메인 구현 에이전트가 책임짐

**데이터 흐름**
- 기획 문서/디자인 메모 -> 라우트 구조 파악 -> 공용 프리미티브 정의 -> 페이지 데이터 구조 수정 -> 컴포넌트 반영 -> lint/실행 확인 -> 문서 업데이트

#### B. Cursor 같은 IDE 보조 AI 활용 전략

**페르소나**
- "빠른 실험을 돕는 UI 페어 프로그래머"

**핵심 전략**
- 전체 구조를 바꾸는 용도가 아니라, 특정 카드/버튼/텍스트/간격 같은 미세 조정에 사용한다.
- 디자이너 감각이 많이 들어가는 세부 영역에서 빠르게 여러 시안을 비교하는 데 활용한다.

**주요 담당 작업**
- Tailwind 클래스 미세 조정
- 문구 톤 변경
- 반복되는 마크업 생성
- 단일 컴포넌트의 시각적 변형 실험

**제약 사항**
- 아키텍처, 공용 레이어, 라우팅 같은 전역 변경은 맡기지 않는다.
- 큰 문맥 없이 단편 수정하다 보면 제품 일관성이 깨질 수 있으므로, 최종 반영 전 메인 에이전트 검토가 필요하다.

**데이터 흐름**
- 현재 열려 있는 파일 컨텍스트 -> 컴포넌트 단위 제안 -> 개발자 확인 -> 메인 브랜치 반영 여부 결정

#### C. Claude 계열 장문 추론 도구 활용 전략

**페르소나**
- "서비스 기획/UX 전략 리뷰어"

**핵심 전략**
- 코드 생성보다 문제 정의, 사용자 여정, 정보 구조, 대안 비교에 사용한다.
- 특히 학생 탭과 선생님 탭의 역할 차이, 한국사 서비스다운 질감, 운영툴다운 긴장감을 문장으로 명확히 고정하는 데 활용한다.

**주요 담당 작업**
- 화면 역할 정의
- 탭별 컨셉 명세
- 테마 DNA 문서 초안
- 제품 북극성 문서 작성
- 디자인 방향의 장단점 분석

**제약 사항**
- 실제 구현 반영 전에는 반드시 코드 레벨 현실성 검토가 필요하다.
- 텍스트상 멋진 방향이 실제 UI 프리미티브 구조와 맞지 않을 수 있으므로, 구현 단계에서는 Codex 계열 도구가 재해석해야 한다.

**데이터 흐름**
- 문제 정의 -> 구조적 인사이트 도출 -> 명세서/문서 작성 -> 구현 에이전트가 실현 가능한 수준으로 재해석

#### D. 도구 간 협업 원칙

도구를 무작정 병렬로 쓰지 않고, 아래와 같이 단계적으로 사용한다.

1. **문제 정의/기획**
   - 장문 추론 도구로 사용자 문제와 화면 역할을 정리
2. **구조 설계**
   - 메인 코딩 에이전트가 실제 파일 구조와 연결해 컴포넌트 단위로 설계
3. **세부 구현**
   - 메인 코딩 에이전트가 반영, IDE 보조 AI는 작은 수정 지원
4. **검증/회귀 확인**
   - lint, 서버 확인, 경로 응답, 화면 점검
5. **문서화**
   - 구현 결과를 다시 제출 문서와 메모리 문서에 반영

즉, 각 AI 도구는 같은 일을 중복해서 하는 것이 아니라, **기획 -> 구현 -> 검증 -> 문서화**의 서로 다른 단계에서 역할을 분담한다.

### 2-3. 토큰 낭비를 최소화하고 유지보수성 및 재현성을 높이기 위한 전략이 있다면 작성해주세요.

이 프로젝트에서 토큰 낭비를 줄이고 유지보수성과 재현성을 높이기 위한 전략은 매우 중요하다. 이유는 이 서비스가 단일 페이지가 아니라 학생용 학습 루프와 선생님용 운영 루프가 동시에 존재하는 구조이기 때문이다. 화면 수가 많아질수록 매번 전체 문맥을 다시 설명하거나, 페이지별로 비슷한 UI를 반복 생성하면 비용과 품질 모두 빠르게 나빠진다.

#### 1. 공용 프리미티브 우선 전략

가장 중요한 전략은 **페이지를 먼저 만들지 않고, 공용 프리미티브를 먼저 추출하는 것**이다. 실제로 현재 프로젝트는 `history-ui`와 `admin`용 프리미티브를 기반으로 페이지를 구성하고 있다. 이는 다음과 같은 장점이 있다.

- 같은 버튼, 카드, 라벨, 보관 패널을 매번 새로 설명할 필요가 없다.
- 하나의 컴포넌트를 개선하면 여러 화면이 동시에 개선된다.
- AI에게도 "이 화면은 새로운 디자인을 만드는 것이 아니라 기존 프리미티브 조합"이라고 명확히 전달할 수 있다.

즉, 토큰을 많이 쓰는 이유 중 하나인 "반복 설명"을 컴포넌트 체계 자체로 줄이는 전략이다.

#### 2. 페이지 역할 문서화 전략

두 번째는 **화면 역할을 문서로 먼저 잠그는 전략**이다. 현재 프로젝트에는 `product-ui-north-star`, `student-ui-north-star`, `admin-memory`, `today-memory`, `history-learning-os-theme-dna-system-v1` 같은 문서가 존재한다. 이 문서들은 단순 참고용이 아니라, AI에게 매번 긴 설명을 다시 하지 않도록 하는 기준서 역할을 한다.

예를 들어:
- `/today`는 시대 입구
- `/notes`는 기억 보관실
- `/review`는 회복실
- `/solve`는 판단실
- `/admin/students`는 학생 개입 운영실

이런 식으로 역할이 고정되어 있으면, 이후 수정 시에도 "이 화면은 무엇을 하는가"를 처음부터 다시 쓰지 않아도 된다. 이는 토큰 절약뿐 아니라, 결과물 일관성 유지에도 매우 중요하다.

#### 3. 데이터 구조와 화면 구조를 분리하는 전략

세 번째는 **페이지 데이터 구조와 프레젠테이션 구조를 분리하는 것**이다. 현재 `/today` 같은 화면은 `pageData`처럼 데이터 모델을 먼저 두고, 그 위에 섹션별 UI를 얹는 방식으로 구성되어 있다. 이런 방식은 다음과 같은 장점이 있다.

- 카피만 수정할 때 전체 JSX를 다시 생성할 필요가 없다.
- AI가 전체 페이지를 통째로 다시 쓰지 않고, 특정 데이터 필드만 수정할 수 있다.
- 나중에 백엔드 실제 API와 연결할 때도 맵핑 포인트가 분명하다.

즉, **화면 하드코딩 최소화**는 토큰 절약과 재현성 확보를 동시에 만족시키는 전략이다.

#### 4. 작업 단위 축소 전략

네 번째는 **작업을 "페이지 전체"가 아니라 "명확한 블록 단위"로 쪼개는 것**이다. 예를 들어,

- Hero 블록
- Time Rail 블록
- Flow Steps 블록
- Chain Cards 블록
- Archive 블록

이렇게 자르면 AI에게도 "이번 작업은 Time Rail만 수정"처럼 좁은 범위를 줄 수 있다. 그러면 긴 파일 전체를 매번 다시 전달하지 않아도 되고, 회귀 버그도 줄어든다.

#### 5. 고성능 모델과 경량 모델의 역할 분리

다섯 번째는 **모든 작업에 최고 모델을 쓰지 않는 전략**이다.

- 구조 설계, 공용화, 복잡한 디버깅: 고성능 모델
- 반복 마크업 수정, 문장 다듬기, 클래스 변경: 경량 모델 또는 IDE 보조

이렇게 역할을 나누면 토큰 비용을 줄이면서도, 중요한 의사결정에는 충분한 추론 품질을 확보할 수 있다.

#### 6. 검증 루틴 고정 전략

여섯 번째는 **검증 과정을 템플릿화하는 것**이다. 현재 프로젝트는 코드 수정 후 다음 루틴을 반복한다.

1. lint 실행
2. 로컬 서버 응답 확인
3. 주요 라우트 접속 확인
4. 겹침/도킹/배경 누수 같은 시각 회귀 체크

이렇게 검증 루틴을 고정하면, AI가 새 변경을 할 때마다 "무엇을 확인해야 하는지"를 다시 길게 설명하지 않아도 된다. 이는 유지보수성과 재현성을 크게 높인다.

#### 7. 문서와 구현의 동시 업데이트 전략

일곱 번째는 **코드만 바꾸고 끝내지 않고, 설계 문서와 메모리 문서를 같이 업데이트하는 것**이다. 프로젝트가 커질수록 "왜 이렇게 만들었는가"가 사라지면 다음 수정 때 같은 논의를 반복하게 된다. 그래서 이번 프로젝트처럼 테마 DNA, 탭 역할, 운영실 구조 같은 내용을 문서로 고정해두면, 다음 사람이 작업하더라도 같은 언어를 쓸 수 있다.

#### 종합 정리

토큰 절약과 유지보수성 향상을 위한 핵심은 다음 세 문장으로 요약할 수 있다.

1. **반복 설명을 줄이기 위해 공용 컴포넌트와 역할 문서를 먼저 만든다.**
2. **전체 페이지를 매번 다시 쓰지 않고, 데이터 구조와 블록 단위로 수정한다.**
3. **구현 후에는 검증 루틴과 문서 업데이트까지 같이 고정해 재현성을 확보한다.**

결국 이 프로젝트에서 AI를 잘 쓰는 방법은 "많이 쓰는 것"이 아니라, **제품 구조가 흔들리지 않도록 AI의 작업 단위를 설계하는 것**이라고 볼 수 있다.

  - 한국사 학습 OS 테마 DNA
